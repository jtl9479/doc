## 🌟 실생활 비유로 이해하기

### 비유 1: 개발 환경 = 요리 도구

```
🍳 요리를 하려면...

조리 도구              →  개발 도구
──────────────────────────────────
가스레인지             →  JDK (Java 실행 환경)
칼, 도마               →  IDE (코드 편집기)
레시피                 →  Java 코드
완성된 요리            →  실행되는 프로그램

도구 없이는 요리 불가능 = JDK 없이는 개발 불가능!
```

### 비유 2: JDK vs JRE

```
🏗️ 건축 비유

JRE (Java Runtime Environment)
├─ 완성된 집에 살기만 하는 사람
├─ Java 프로그램을 "실행"만 가능
└─ 일반 사용자용

JDK (Java Development Kit)
├─ 집을 직접 짓는 건축가
├─ Java 프로그램을 "개발 + 실행" 가능
├─ JRE를 포함 (JDK > JRE)
└─ 개발자용 ⭐

개발자는 반드시 JDK 설치!
```

### 비유 3: IDE = 워드프로세서

```
📝 글쓰기 도구

메모장              →  일반 텍스트 편집기
                      - 기능 없음
                      - 불편함

MS Word            →  IDE (IntelliJ, Eclipse)
                      - 맞춤법 검사 (문법 오류)
                      - 자동 완성
                      - 서식 지정
                      - 훨씬 편리!

메모장으로도 코딩 가능하지만, IDE가 100배 편함!
```

### 비유 4: 환경 변수 = 주소록

```
📱 스마트폰 주소록

환경 변수 없을 때      →  주소록 없이 전화
──────────────────────────────────────────
매번 전화번호 외워야 함  →  매번 Java 경로 입력
   "010-1234-5678"        "C:\Program Files\Java\..."

환경 변수 설정 후      →  주소록에 저장
──────────────────────────────────────────
이름만 검색하면 OK!     →  "java" 명령어만 입력
   "엄마" 클릭            어디서든 바로 실행!

💡 환경 변수 = 컴퓨터의 주소록
   한 번 설정하면 계속 편리하게 사용!
```

### 비유 5: 개발 환경 = 게임 캐릭터 장비

```
🎮 게임 캐릭터 육성

JDK                →  기본 스탯 (레벨, HP, MP)
                      - 없으면 게임 시작 불가
                      - 버전 = 레벨

IDE                →  강력한 무기/방어구
                      - 일반 공격 → 스킬 콤보
                      - 생존율 대폭 증가
                      - IntelliJ = 전설 등급 무기 ⚔️

플러그인           →  악세서리/아이템
                      - 능력치 추가 상승
                      - 편의 기능 제공

┌─────────────────────────────────────┐
│ 초보 개발자 (장비 없음)             │
│ - 메모장 + 명령어로만 개발          │
│ - 생산성: ⭐☆☆☆☆                  │
│ - 오류율: 높음 ↑                    │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 고수 개발자 (완벽한 장비)           │
│ - JDK 17 + IntelliJ + 플러그인      │
│ - 생산성: ⭐⭐⭐⭐⭐             │
│ - 오류율: 낮음 ↓                    │
└─────────────────────────────────────┘

💡 같은 실력이라도 장비(개발 환경)가 결과를 좌우!
```

## 📖 필요한 개발 도구 개요

### 1️⃣ JDK (Java Development Kit) - 필수!

```
🔧 JDK = Java 개발 키트

포함 내용:
├─ javac (컴파일러)      .java → .class 변환
├─ java (실행기)         .class 파일 실행
├─ jar (압축 도구)       여러 파일 묶기
├─ javadoc (문서 생성)   API 문서 만들기
└─ 기타 개발 도구들

버전 선택:
├─ Java 8 (LTS)     가장 많이 쓰임 (레거시)
├─ Java 11 (LTS)    안정적
├─ Java 17 (LTS)    추천! ⭐⭐⭐
└─ Java 21 (LTS)    최신 (2023)

💡 LTS = Long Term Support (장기 지원)
```

### 2️⃣ IDE (통합 개발 환경) - 강력 추천!

```
🖥️ IDE 선택지

IntelliJ IDEA        ⭐⭐⭐ 1순위 추천!
├─ 장점: 강력한 자동 완성, 편리한 디버깅
├─ 단점: 약간 무거움
├─ 버전: Community (무료), Ultimate (유료)
└─ 사용 기업: 카카오, 네이버, 배민 등 대부분

Eclipse              ⭐⭐ 2순위
├─ 장점: 무료, 가벼움, 오래된 역사
├─ 단점: IntelliJ보다 기능 부족
└─ 사용: 중소기업, 학교

VS Code + Extension  ⭐ 입문자용
├─ 장점: 가볍고 빠름
├─ 단점: Java 전용 아님 (다목적)
└─ 추천: 입문자, 간단한 프로젝트

NetBeans             잘 안 쓰임
```

## 💻 실습 1: JDK 설치하기

### Step 1: JDK 다운로드

**추천 배포판:**

```
1. Oracle JDK         공식, 상업적 사용 시 유료
2. OpenJDK           완전 무료 ⭐ 추천!
   ├─ Adoptium (Eclipse Temurin)
   ├─ Amazon Corretto
   └─ Azul Zulu

💡 이 가이드에서는 Adoptium 사용
```

**다운로드 방법:**

```
1. https://adoptium.net 접속
2. 'Temurin 17 (LTS)' 선택
3. 운영체제 선택
   - Windows: .msi 설치 파일
   - macOS: .pkg 설치 파일
   - Linux: .tar.gz 압축 파일
4. 다운로드 클릭
```

### Step 2: Windows 설치

```
📌 Windows 설치 과정

1. 다운로드한 .msi 파일 실행
2. "Next" 클릭
3. 설치 경로 확인 (기본값 추천)
   C:\Program Files\Eclipse Adoptium\jdk-17.x.x
4. "Set JAVA_HOME" 체크 ✅ (중요!)
5. "Add to PATH" 체크 ✅ (중요!)
6. "Install" 클릭
7. 설치 완료 대기 (2-3분)
8. "Finish" 클릭
```

**설치 확인:**

```cmd
# 명령 프롬프트(cmd) 실행
# 방법 1: Win + R → cmd 입력
# 방법 2: 시작 메뉴에서 "명령 프롬프트" 검색

java -version
```

**예상 출력:**

```
openjdk version "17.0.9" 2023-10-17
OpenJDK Runtime Environment Temurin-17.0.9+9 (build 17.0.9+9)
OpenJDK 64-Bit Server VM Temurin-17.0.9+9 (build 17.0.9+9, mixed mode)
```

**환경 변수 수동 설정 (자동 설정 안 된 경우):**

```
1. 시작 메뉴 → "환경 변수" 검색
2. "시스템 환경 변수 편집" 클릭
3. "환경 변수" 버튼 클릭
4. 시스템 변수 섹션에서 "새로 만들기"
   - 변수 이름: JAVA_HOME
   - 변수 값: C:\Program Files\Eclipse Adoptium\jdk-17.x.x
5. Path 변수 편집
   - "편집" 클릭
   - "새로 만들기" 클릭
   - %JAVA_HOME%\bin 추가
6. 모두 "확인" 클릭
7. cmd 재시작 후 확인
```

### Step 3: macOS 설치

```
📌 macOS 설치 과정

방법 1: 설치 파일 (.pkg)
1. 다운로드한 .pkg 파일 실행
2. 설치 지침 따라하기
3. 비밀번호 입력
4. 설치 완료

방법 2: Homebrew (추천!) ⭐
$ brew install openjdk@17
$ sudo ln -sfn /opt/homebrew/opt/openjdk@17/libexec/openjdk.jdk \
  /Library/Java/JavaVirtualMachines/openjdk-17.jdk
```

**환경 변수 설정:**

```bash
# ~/.zshrc 파일 편집 (또는 ~/.bash_profile)
$ nano ~/.zshrc

# 다음 라인 추가
export JAVA_HOME=$(/usr/libexec/java_home -v 17)
export PATH=$JAVA_HOME/bin:$PATH

# 저장 후 적용
$ source ~/.zshrc
```

**설치 확인:**

```bash
$ java -version
openjdk version "17.0.9" 2023-10-17
```

### Step 4: Linux (Ubuntu) 설치

```
📌 Ubuntu/Debian 설치

$ sudo apt update
$ sudo apt install openjdk-17-jdk

# 설치 확인
$ java -version
$ javac -version
```

**환경 변수 설정:**

```bash
# ~/.bashrc 파일 편집
$ nano ~/.bashrc

# 다음 라인 추가
export JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64
export PATH=$JAVA_HOME/bin:$PATH

# 적용
$ source ~/.bashrc
```

## 💻 실습 3: 첫 Java 프로젝트 만들기

### Step 1: 새 프로젝트 생성

```
📁 프로젝트 생성 과정

1. IntelliJ 시작 화면에서 "New Project" 클릭

2. 프로젝트 설정
   ┌────────────────────────────────┐
   │ Name: HelloJava               │
   │ Location: C:\Users\...\       │
   │ Language: Java ⭐             │
   │ Build System: IntelliJ        │
   │ JDK: 17 (자동 감지)           │
   │ Add sample code: ✅ 체크      │
   └────────────────────────────────┘

3. "Create" 클릭

4. 프로젝트 구조 확인
   HelloJava/
   ├── .idea/           (IntelliJ 설정)
   ├── src/             (소스 코드 폴더)
   │   └── Main.java
   └── HelloJava.iml    (프로젝트 설정)
```

### Step 2: Hello World 작성

```java
// src/Main.java
public class Main {
    public static void main(String[] args) {
        // TODO: Hello World 출력
        System.out.println("Hello, World!");
        System.out.println("나의 첫 Java 프로그램!");

        // 간단한 계산
        int a = 10;
        int b = 20;
        System.out.println("10 + 20 = " + (a + b));
    }
}
```

### Step 3: 프로그램 실행

## ⚠️ 트러블슈팅: 흔한 설치 오류

### 오류 1: 'java'은(는) 내부 또는 외부 명령이 아닙니다

```
❌ 증상 (Windows)
C:\> java -version
'java'은(는) 내부 또는 외부 명령, 실행할 수 있는
프로그램, 또는 배치 파일이 아닙니다.

🔍 원인:
환경 변수 PATH에 Java가 등록 안 됨

✅ 해결 방법:
1. JDK 설치 경로 확인
   C:\Program Files\Eclipse Adoptium\jdk-17.x.x\bin

2. 환경 변수 설정
   - 시스템 환경 변수 편집
   - Path에 위 경로 추가

3. cmd 재시작 (중요!)
   - 기존 cmd 창 닫기
   - 새로 열기

4. 다시 확인
   C:\> java -version
   ✅ 성공!
```

### 오류 2: JDK 버전이 다르게 나옴

```
❌ 증상
설치한 버전: Java 17
확인한 버전: Java 8 or 11

🔍 원인:
여러 JDK가 설치되어 있고, 잘못된 버전이 PATH에 먼저 등록됨

✅ 해결 방법 (Windows):
1. 설치된 Java 확인
   C:\> where java
   C:\Program Files\Java\jdk1.8.0\bin\java.exe
   C:\Program Files\Eclipse Adoptium\jdk-17\bin\java.exe

2. PATH 순서 조정
   - 환경 변수 편집
   - Java 17 경로를 맨 위로 이동

3. 또는 기존 Java 제거
   - 제어판 → 프로그램 제거
   - 사용하지 않는 JDK 삭제

4. 확인
   C:\> java -version
   openjdk version "17.0.9" ✅
```

## 🛠️ 실전 프로젝트: 개발 환경 체크 프로그램

### 프로젝트 목표

```
현재 개발 환경을 점검하는 Java 프로그램 만들기
- JDK 버전 확인
- 운영체제 확인
- 사용 가능 메모리 확인
- 환경 변수 확인
```

### 구현 코드

```java
// EnvironmentChecker.java
public class EnvironmentChecker {
    public static void main(String[] args) {
        System.out.println("=".repeat(50));
        System.out.println("        개발 환경 체크 프로그램");
        System.out.println("=".repeat(50));
        System.out.println();

        // Java 정보
        System.out.println("📌 Java 정보");
        System.out.println("Java 버전: " +
            System.getProperty("java.version"));
        System.out.println("Java 벤더: " +
            System.getProperty("java.vendor"));
        System.out.println("Java 홈: " +
            System.getProperty("java.home"));
        System.out.println();

        // 운영체제 정보
        System.out.println("💻 운영체제 정보");
        System.out.println("OS 이름: " +
            System.getProperty("os.name"));
        System.out.println("OS 버전: " +
            System.getProperty("os.version"));
        System.out.println("OS 아키텍처: " +
            System.getProperty("os.arch"));
        System.out.println();

        // 사용자 정보
        System.out.println("👤 사용자 정보");
        System.out.println("사용자 이름: " +
            System.getProperty("user.name"));
        System.out.println("홈 디렉토리: " +
            System.getProperty("user.home"));
        System.out.println("현재 디렉토리: " +
            System.getProperty("user.dir"));
        System.out.println();

        // 메모리 정보
        System.out.println("🧠 메모리 정보");
        Runtime runtime = Runtime.getRuntime();
        long maxMemory = runtime.maxMemory();
        long totalMemory = runtime.totalMemory();
        long freeMemory = runtime.freeMemory();

        System.out.println("최대 메모리: " +
            (maxMemory / 1024 / 1024) + " MB");
        System.out.println("할당된 메모리: " +
            (totalMemory / 1024 / 1024) + " MB");
        System.out.println("여유 메모리: " +
            (freeMemory / 1024 / 1024) + " MB");
        System.out.println();

        // CPU 정보
        System.out.println("⚡ CPU 정보");
        System.out.println("사용 가능한 프로세서: " +
            runtime.availableProcessors() + "개");
        System.out.println();

        // 환경 변수 확인
        System.out.println("🔧 주요 환경 변수");
        String javaHome = System.getenv("JAVA_HOME");
        String path = System.getenv("PATH");

        if (javaHome != null) {
            System.out.println("JAVA_HOME: " + javaHome);
        } else {
            System.out.println("⚠️ JAVA_HOME이 설정되지 않았습니다!");
        }

        if (path != null && path.contains("java")) {
            System.out.println("✅ PATH에 Java가 포함되어 있습니다.");
        } else {
            System.out.println("⚠️ PATH에 Java가 없습니다!");
        }

        System.out.println();
        System.out.println("=".repeat(50));
        System.out.println("        환경 체크 완료!");
        System.out.println("=".repeat(50));
    }
}
```

```

sdk install java 11.0.20-tem

sdk use java 17.0.9-tem      # 현재 터미널만

brew install jenv

# 버전 설정
jenv global 17.0      # 전역
jenv local 11.0       # 현재 폴더만

  /Library/Java/JavaVirtualMachines/openjdk-17.jdk

💡 실무 팁:
```

<details>
<summary><strong>1. JDK와 JRE의 차이점을 설명해주세요.</strong></summary>

**모범 답안 포인트**
- JRE는 Java 프로그램을 실행하는 환경
- JDK는 Java 프로그램을 개발하는 도구 모음
- JDK는 JRE를 포함하는 관계

**예시 답변**
> "JRE는 Java Runtime Environment의 약자로, 작성된 Java 프로그램을 실행할 수 있는 환경입니다. 일반 사용자가 Java로 만들어진 애플리케이션을 실행할 때 필요합니다.
>
> JDK는 Java Development Kit의 약자로, Java 프로그램을 개발하기 위한 도구 모음입니다. javac 컴파일러, jar 도구 등이 포함되어 있으며, JRE도 포함하고 있습니다. 따라서 개발자는 JDK를 설치해야 합니다."

**꼬리 질문**
- Q: 그럼 개발자는 왜 JRE만 설치하면 안 되나요?
- A: JRE에는 javac 컴파일러가 없어서 .java 파일을 .class 파일로 컴파일할 수 없기 때문입니다.

**실무 연관**
- 실무에서 서버에는 JRE만 설치하고, 개발 PC에는 JDK를 설치하는 경우가 많습니다.
- 경량화와 보안을 위해 프로덕션 환경에는 불필요한 개발 도구를 제외한 JRE만 배포합니다.

<details>
<summary><strong>1. JVM, JRE, JDK의 관계와 각각의 역할을 설명해주세요.</strong></summary>

**모범 답안 포인트**
- JVM: Java 바이트코드를 실행하는 가상 머신
- JRE: JVM + 표준 라이브러리
- JDK: JRE + 개발 도구
- 포함 관계: JDK > JRE > JVM

**예시 답변**
> "JVM(Java Virtual Machine)은 자바 바이트코드(.class)를 실제 하드웨어에서 실행할 수 있도록 변환하는 가상 머신입니다. 플랫폼 독립성을 제공하는 핵심 요소입니다.
>
> JRE(Java Runtime Environment)는 JVM과 Java 표준 라이브러리(java.lang, java.util 등)를 포함하는 실행 환경입니다. Java 애플리케이션을 실행하는 데 필요한 모든 것이 들어 있습니다.
>
> JDK(Java Development Kit)는 JRE에 개발 도구들(javac 컴파일러, javadoc, jar 등)을 추가한 개발 키트입니다. 포함 관계로 보면 JDK ⊃ JRE ⊃ JVM입니다."

**실무 예시**
```bash
JDK (개발자 PC)
├── bin/
│   ├── javac (컴파일러)
│   ├── java (실행기)
│   └── jar (패키징)
└── jre/
    ├── lib/ (표준 라이브러리)
    └── bin/
        └── JVM (가상 머신)

JRE (프로덕션 서버)
├── lib/ (표준 라이브러리)
└── bin/
    └── JVM (가상 머신)
```

**꼬리 질문**
- Q: 프로덕션 환경에 JDK 대신 JRE만 배포하는 이유는?
- A: 보안과 경량화 때문입니다. 컴파일러 등 개발 도구가 서버에 노출되면 보안 위험이 있고, 불필요한 도구로 인해 이미지 크기가 커집니다.

**실무 연관**
- Docker 이미지 최적화 시 JRE 기반 이미지를 사용
- GraalVM Native Image로 JVM 없이 실행 파일 생성 가능

</details>

<summary><strong>2. 여러 프로젝트에서 서로 다른 Java 버전을 사용할 때 어떻게 관리하나요?</strong></summary>

**모범 답안 포인트**
- SDKMAN, jEnv 같은 버전 관리 도구 사용
- IDE에서 프로젝트별 JDK 설정
- Docker 컨테이너로 환경 격리
- Gradle/Maven Toolchain 활용

**예시 답변**
> "프로젝트별로 다른 Java 버전을 사용해야 할 때는 여러 전략을 사용할 수 있습니다.
>
> 첫째, SDKMAN이나 jEnv 같은 버전 관리 도구를 사용합니다. 프로젝트 폴더에 `.sdkmanrc` 파일을 두면 폴더 진입 시 자동으로 해당 Java 버전으로 전환됩니다.
>
> 둘째, IntelliJ에서는 프로젝트별로 다른 SDK를 설정할 수 있습니다. File → Project Structure에서 프로젝트마다 독립적인 JDK를 지정할 수 있습니다.
>
> 셋째, Docker를 사용하여 각 프로젝트를 완전히 격리된 환경에서 실행할 수 있습니다. 넷째, Gradle의 Toolchain 기능을 사용하면 빌드 시 특정 JDK 버전을 자동으로 다운로드하여 사용합니다."

**실무 예시**
```groovy
// build.gradle - Gradle Toolchain
java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(17)
    }
}

// .sdkmanrc - 프로젝트별 자동 전환
java=17.0.9-tem
```

**꼬리 질문**
- Q: CI/CD 파이프라인에서는 어떻게 관리하나요?
- A: Docker 이미지에 특정 JDK 버전을 포함시키거나, GitHub Actions의 setup-java 액션으로 필요한 버전을 지정합니다.

**실무 연관**
- 마이크로서비스 환경에서 각 서비스가 다른 Java 버전 사용 가능
- 레거시 시스템과 신규 시스템을 동시에 개발할 때 필수

<summary><strong>3. OpenJDK와 Oracle JDK의 차이점과 선택 기준을 설명해주세요.</strong></summary>

**모범 답안 포인트**
- 기술적으로는 거의 동일
- 라이선스 차이: OpenJDK는 완전 무료
- Oracle JDK는 상업용 사용 시 유료
- 기업에서는 OpenJDK 배포판(Adoptium, Corretto) 선호

**예시 답변**
> "Java 11부터 Oracle JDK와 OpenJDK는 기술적으로 거의 동일합니다. 둘 다 같은 소스 코드 베이스를 사용하며, 성능과 기능 면에서 차이가 거의 없습니다.
>
> 주요 차이점은 라이선스입니다. OpenJDK는 GPL 라이선스로 완전 무료이며 상업적 사용도 가능합니다. 반면 Oracle JDK는 개인/학습용은 무료이지만, 상업적으로 사용하려면 유료 구독이 필요합니다.
>
> 실무에서는 Adoptium(Eclipse Temurin), Amazon Corretto, Azul Zulu 같은 OpenJDK 기반 배포판을 많이 사용합니다. 이들은 무료이면서도 장기 지원을 제공하기 때문입니다."

**실무 예시**
```
배포판 선택 기준:

Adoptium (Eclipse Temurin)
✅ 가장 많이 사용됨
✅ 커뮤니티 지원 강력
✅ 모든 플랫폼 지원

Amazon Corretto
✅ AWS 최적화
✅ Amazon 지원
✅ 무료 장기 지원

Azul Zulu
✅ 엔터프라이즈 지원
✅ 임베디드 시스템 지원
```

**꼬리 질문**
- Q: 회사에서 Oracle JDK를 사용하면 반드시 비용을 내야 하나요?
- A: 프로덕션 환경에서 Oracle JDK를 사용하면 Oracle Java SE 구독 비용을 지불해야 합니다. 하지만 개발/테스트 환경은 무료입니다.

**실무 연관**
- 대부분의 기업이 라이선스 비용 절감을 위해 OpenJDK로 전환했습니다.
- 쿠버네티스, 클라우드 환경에서는 OpenJDK 기반 경량 이미지 사용

<summary><strong>4. 개발 환경을 Docker로 구성하는 이유와 장단점을 설명해주세요.</strong></summary>

**모범 답안 포인트**
- 환경 일관성 보장
- "내 컴퓨터에서는 되는데" 문제 해결
- 빠른 온보딩
- 단점: 학습 곡선, 리소스 사용

**예시 답변**
> "Docker를 사용하면 개발, 테스트, 프로덕션 환경을 동일하게 유지할 수 있습니다. 컨테이너에 Java 버전, 라이브러리, 설정 등을 모두 포함시켜 '내 컴퓨터에서는 되는데 서버에서는 안 돼'라는 문제를 원천적으로 방지합니다.
>
> 또한 신입 개발자가 환경 설정에 시간을 낭비하지 않고 `docker-compose up` 한 번으로 모든 개발 환경을 구성할 수 있어 온보딩이 빠릅니다.
>
> 단점은 Docker 자체를 학습해야 하는 초기 비용과, 컨테이너 실행으로 인한 추가 리소스 사용입니다. 하지만 장점이 훨씬 크기 때문에 현대 개발 환경에서는 거의 필수입니다."

**실무 예시**
```dockerfile
# Dockerfile
FROM eclipse-temurin:17-jdk-alpine
WORKDIR /app
COPY . .
RUN ./gradlew build
CMD ["java", "-jar", "app.jar"]
```

```yaml
# docker-compose.yml
version: '3'
services:
  app:
    build: .
    ports:
      - "8080:8080"
    environment:
      - JAVA_OPTS=-Xmx512m
  db:
    image: postgres:15
    environment:
      - POSTGRES_DB=myapp
```

**꼬리 질문**
- Q: Docker Desktop 대신 다른 대안이 있나요?
- A: Podman, Rancher Desktop, Colima 등이 있습니다. 특히 Podman은 Docker와 호환되면서 rootless로 실행할 수 있어 보안상 유리합니다.

**실무 연관**
- 마이크로서비스 아키텍처에서 필수
- CI/CD 파이프라인에서 Docker 이미지로 배포
- 로컬에서 Kubernetes 환경 테스트 가능

<summary><strong>5. 팀의 코드 스타일과 개발 환경 설정을 어떻게 통일하나요?</strong></summary>

**모범 답안 포인트**
- EditorConfig로 기본 설정 통일
- Checkstyle, SpotBugs 같은 정적 분석 도구
- IDE 설정 파일 공유
- pre-commit hook으로 강제
- 코드 리뷰에서 확인

**예시 답변**
> "팀의 코드 스타일을 통일하기 위해 여러 방법을 사용합니다.
>
> 첫째, `.editorconfig` 파일로 들여쓰기, 줄바꿈 문자, 인코딩 등 기본 설정을 통일합니다. 대부분의 IDE가 이를 자동으로 인식합니다.
>
> 둘째, Checkstyle이나 Google Java Format 같은 코드 스타일 검사 도구를 프로젝트에 포함시킵니다. Gradle/Maven 빌드 시 자동으로 검사하도록 설정합니다.
>
> 셋째, IntelliJ 설정 파일을 Git에 포함시켜 팀원 모두가 같은 IDE 설정을 사용하도록 합니다. `.idea/codeStyles` 폴더를 커밋합니다.
>
> 넷째, Git pre-commit hook을 설정하여 스타일 위반 코드는 커밋되지 않도록 강제합니다."

**실무 예시**
```
// .editorconfig
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true

[*.java]
indent_style = space
indent_size = 4
```

```groovy
// build.gradle - Checkstyle 적용
plugins {
    id 'checkstyle'
}

checkstyle {
    toolVersion = '10.12.0'
    configFile = file("${rootDir}/config/checkstyle/checkstyle.xml")
}
```

- Q: 코드 포맷팅을 자동화하는 방법은?

**실무 연관**
- 코드 리뷰 시간을 줄이고 본질적인 로직 검토에 집중
- 신입 개발자도 일관된 코드를 작성할 수 있도록 도움
- 오픈소스 프로젝트 기여 시 필수
